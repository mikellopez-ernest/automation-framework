# Purpose

This document describes how the Automation Framework interacts with the Dinantia Student Tracking module.

Rather than documenting the implementation itself, this document explains the behaviour of the Dinantia application, the observations made during development and the design decisions adopted to make the automation reliable.

---

# Scope

This document covers:

* Navigation through the Tracking module.
* School year selection.
* Detail view.
* Report export.
* Known application behaviour.
* Reliability considerations.

This document does not describe browser infrastructure or authentication.

---

# Functional Overview

The automation exports the detailed tracking report for a selected school year.

The complete workflow is:

```text
Restore session
        │
        ▼
Open Inbox
        │
        ▼
Open Tracking
        │
        ▼
Select School Year
        │
        ▼
Wait for Chart Data
        │
        ▼
Open Detail View
        │
        ▼
Export Report
        │
        ▼
Download Excel File
```

The workflow has been designed to tolerate temporary failures while minimizing unnecessary browser interactions.

---

# Navigation

The Tracking module is accessed from the authenticated interface.

Navigation is performed using stable semantic selectors whenever possible.

The automation intentionally avoids selectors based on dynamically generated identifiers.

Preferred selector types include:

* accessible roles;
* placeholder text;
* visible text;
* semantic HTML structure.

Dynamic identifiers generated by the application should not be used unless no alternative exists.

---

# School Year Selection

The selected school year is controlled through a dropdown component.

Before interacting with the selector, the automation verifies the currently selected value.

If the requested school year is already selected, no additional interaction is performed.

This optimization reduces browser activity and avoids unnecessary network requests.

---

# Chart Data Loading

Changing the selected school year triggers one or more asynchronous requests.

The automation waits for the application to finish loading before continuing.

During development it was observed that the application may issue multiple requests to the same endpoint.

Example:

```text
POST /web/attitude/get_chart_data
```

A typical execution produces:

Response 1

* Status: 200
* Records: 0

Response 2

* Status: 200
* Records: 20474

The first response frequently represents an intermediate application state rather than the final dataset.

For this reason, the framework evaluates all matching responses and uses the valid response containing the largest dataset.

This strategy has proven significantly more reliable than assuming a fixed number of requests.

---

# Detail View

The chart view does not provide the downloadable report.

After the chart data has finished loading, the automation opens the **Detail** view.

Navigation is considered complete only after the browser reaches the corresponding URL.

This avoids race conditions where the Detail page has not yet finished loading.

---

# Report Export

The Detail view exposes the report export functionality.

Selecting **Export** triggers a browser download.

The framework waits for the actual download event rather than relying on fixed delays.

Downloads are handled through the reusable DownloadManager component.

---

# Temporary Server Errors

During development an intermittent server-side behaviour was identified.

Occasionally, pressing **Export** opens a temporary browser tab displaying an HTTP 500 error instead of generating the report.

In most observed cases:

* the first export fails;
* closing the temporary tab;
* retrying the export immediately;

results in a successful download.

The framework therefore performs automatic retries before reporting a failure.

This behaviour is considered an application characteristic rather than an automation failure.

---

# Download Behaviour

Downloads preserve the filename suggested by Dinantia.

If a file with the same name already exists, a numeric suffix is appended automatically.

Example:

```text
tracking.xlsx
tracking_1.xlsx
tracking_2.xlsx
```

No existing files are overwritten.

---

# Reliability Strategies

The automation intentionally avoids relying on timing assumptions.

Instead it waits for observable application events such as:

* visible elements;
* URL changes;
* network responses;
* browser download events.

This approach significantly improves stability across different execution speeds.

---

# Selector Strategy

The following selector priority is used throughout the project.

1. Accessible roles.
2. Labels.
3. Placeholder text.
4. Visible text.
5. Stable CSS selectors.
6. XPath.

Dynamic identifiers should be avoided whenever possible.

---

# Known Application Characteristics

The following behaviours were observed during development.

## Multiple matching elements

Some pages contain more than one element matching the same selector.

Selectors may therefore require additional context.

---

## Dynamic identifiers

Many input elements use dynamically generated identifiers.

These identifiers change between sessions and should never be used directly.

---

## Multiple asynchronous responses

A single user interaction may generate several network requests representing intermediate states.

The automation should validate the received data rather than assuming the first response is final.

---

## Temporary export failures

The export operation may occasionally fail with an HTTP 500 response.

Automatic retries mitigate this behaviour successfully.

---

# Design Decisions

Several implementation decisions were influenced by the observed behaviour of the application.

These include:

* persistent authenticated sessions;
* validation of loaded sessions;
* semantic selectors;
* response-based synchronization;
* reusable download infrastructure;
* automatic export retries.

Together these decisions prioritize robustness over implementation simplicity.

---

# Future Evolution

Possible future automations within the Tracking module include:

* creating student incidents;
* updating incident status;
* retrieving additional tracking information;
* exporting filtered reports.

Whenever possible, these features should reuse the existing navigation and synchronization mechanisms.
